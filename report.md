# Full Report

## Introduction

This case study analyzes a security scenario from the LetsDefend platform involving suspicious PowerShell activity related to CVE-2024-49138.

The alert suggested that a malicious PowerShell command attempted to download and execute a tool designed to bypass Endpoint Detection and Response (EDR) protections.

This investigation analyzes the full attack chain, including the initial PowerShell activity, malware download, execution behavior, and subsequent system interactions.

The goal of this analysis is to determine whether the alert represents a true compromise and identify indicators of compromise (IOCs) that could be used for future detection.

---

## Alert Overview

- Alert Name: SOC335 - CVE-2024-49138 Exploitation Detected
- Severity: Medium
- Category: Privilege Escalation
- Platform: LetsDefend

The investigation began after an alert indicated suspicious PowerShell activity associated with a possible exploitation attempt.

---

## Investigation Findings

### 1. PowerShell activity

The process observed in the logs was:

`C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe`

PowerShell was responsible for both DNS resolution and file creation events associated with the suspicious activity.

### 2. DNS query to GitHub

The logs showed that PowerShell queried:

`github.com`

Resolved IP:

`140.82.113.4`

This strongly suggested that PowerShell was attempting to retrieve external content from GitHub-hosted infrastructure.

### 3. ZIP download

PowerShell then created:

`C:\Users\LetsDefend\Downloads\EDR-Freeze_1.0.zip`

This indicates that the payload was first downloaded as a ZIP archive.

### 4. Execution policy evaluation

A temporary PowerShell file was later created:

`C:\Users\LetsDefend\AppData\Local\Temp\_PSScriptPolicyTest_*.ps1`

This file is typically generated by PowerShell when evaluating execution policy behavior. It should not be treated by itself as the malicious payload, but it is relevant to the execution sequence.

### 5. Payload execution

The downloaded executable was:

`C:\Users\LetsDefend\Downloads\EDR-Freeze_1.0.exe`

This executable was identified as EDRFreeze, a tool commonly associated with attempts to interfere with endpoint security controls.

### 6. VirusTotal analysis

SHA256:

`970c7834e58b6ef22473875167a333dbb33bf7b667d1cb814829f68579cd85f7`

VirusTotal detection showed the file was flagged by multiple vendors as malicious / hacktool-related.

### 7. Child process creation

The payload spawned:

`C:\Windows\System32\WerFaultSecure.exe`

This is a legitimate Windows binary, but in this context its execution was suspicious because it was spawned by EDR-Freeze_1.0.exe.

### 8. Network activity

The host communicated with several external IP addresses during the sequence, including:

- `23.221.245.214`
- `185.199.111.133`
- `23.78.9.173`

This supports the conclusion that the activity was not limited to local file execution and involved outbound communication.

---

## Conclusion

The evidence supports a True Positive conclusion.

The host executed suspicious PowerShell activity that led to GitHub resolution, archive download, execution of the EDRFreeze tool, and spawning of a legitimate Windows process in a suspicious context.

This case demonstrates how attackers can use public infrastructure and legitimate system binaries as part of an evasion-oriented execution chain.
